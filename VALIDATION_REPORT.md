# Validation report: compact Contact-SDF Atlas backend

This report was generated by `python scripts/build_and_validate.py` after adding the compact batch query backend, tricubic scalar SDF baseline, and separated construction/query timing.

## What changed in this version

- Added `feature_max_depth` to `build_adaptive_contact_sdf_atlas(...)`.
- Smooth cells stop at `max_depth`; edge/vertex/multi-normal-sector cells can refine deeper to `feature_max_depth`.
- Added normal-sector-driven feature detection. A triangle edge hit alone is not treated as a physical sharp feature; refinement is triggered by competing corner-normal sectors.
- Sharp-feature validation metrics now use only competing active normal sectors; point-to-triangle edge/vertex regions on smooth benchmarks are excluded from cone-hit statistics.
- Added `feature_enrichment`, which bakes additional candidate normal-sector planes sampled inside feature leaves for sharp/mixed benchmarks.
- The projection backend now exposes welded-edge and welded-vertex incident corner-normal sectors, so the online reference, uniform atlas and adaptive atlas use the same normal-sector convention.
- Added a solver-facing `eval_compact(...)` path that returns fixed-size gap, primary normal, mode, leaf id and candidate count fields without allocating Python candidate lists.
- Added a conventional tricubic scalar SDF baseline using the same sampled signed-distance grid as the trilinear baseline.
- Vectorized the trilinear and tricubic grid baselines so query-time comparisons use batch backends for all non-projection methods.
- Runtime atlas evaluation remains projection-free: lookup through the finest leaf grid plus polynomial/plane evaluation.

## Interpretation

- **Gap accuracy improves on all three bodies** compared with both scalar baselines. Tricubic interpolation improves over trilinear interpolation, but the feature-adaptive atlas still gives the lowest gap RMSE.
- **Normal accuracy improves on all three bodies** compared with trilinear and tricubic SDFs. The sharp benchmarks benefit most because scalar interpolants cannot expose multiple normal sectors.
- **Normal-cone hit rate improves on sharp models**: `hex_prism` improves from 0.677 to 0.923, and `cone` improves from 0.554 to 0.846. This is the main purpose of feature-specific refinement.
- **Construction and query costs are now separated**. Adaptive construction is intentionally more expensive because projection, refinement tests and sector enrichment are offline; compact runtime lookup is sub-microsecond per query in this batch Python backend.
- The ellipsoid is smooth, so cone-hit statistics remain n/a. A small number of adaptive leaves can still be marked as multi-sector records by the discrete corner-normal backend, but they are not counted as physical sharp-feature queries.
- The supplemental validation suite adds sphere exact Hessian, wedge, capped cylinder, cone side/rim/apex, ablations, mesh consistency, noisy normals, torus and rigid-transform invariance. It reports `issues: []`.
